Apply software engineering security principles to AI applications. 
Implement least privilege database access, scope tools properly, inherit authentication context for multi-tenant apps, and build comprehensive logging and monitoring.

Observability Is Non Negotiable
Recommended: Behind the Scenes of AI Observability in Production [ https://substack.com/redirect/c1057f29-303f-4622-ada3-4c7abc0a50a7?j=eyJ1IjoibjBrMyJ9.SQ5ZKHxJh0MocVfrsArKZFhT_ts26OEfWqtevr5N8ms ]
Don’t settle for just logging tool calls and messages. 
Observability is the key to evolving your AI Product and consistently make it more secure and performant. 
The biggest challenges in AI Projects are silent errors. Agents trying to do things they shouldn’t, hallucinations on actions that were executed but nothing happened. 
Take the time to implement Observability and know what’s happening behind the scenes.

When You Want Extra Security... Add Human In The Loop 👀
I haven’t implemented this one production. But there’s nothing better than forcing the agent to ask for permissions, input or extra validation to fully control its behaviour.
There are usually 2 types: 
User Permission
User Input 
Basically you control execution flow and there’s no way the LLM messes anything up unless the user allows it. 
I’ve been evaluating **Safety Operations** over the last months, to see if my agents comply with asking the user before destructive operations (e.g. DELETE REST API endpoints) and it was most of the time going through without asking. 
This Human In The Loop approach is a code driven guardrail that will block execution if rejected, and the agent loop will also “break”. 
I highly recommend exploring this, specially if the impact of destructive operations in your industry is high.
🛡️ Security Layers Summary
Your production-ready AI agent should have these defense layers:
Layer 1: Input Validation
Text classifiers for prompt injection detection
Input sanitization (remove malicious XML/Markdown)
Custom pattern matching for known attacks
Human In The Loop 
Layer 2: Guardrails
Topical guardrails to keep agent on-scope
PII detection and masking
Content moderation for harmful content
LLM-as-judge for request appropriateness
Layer 3: Output Control
Structured outputs with Pydantic schemas
Response format enforcement
Instructor library for automatic retries
Output validation before downstream use
Layer 4: Architecture
Least privilege database access & Scoped tool permissions
Authentication inheritance for multi-tenant
Comprehensive logging, monitoring & Observability Tracking
Cloud content filtering

📝 Final Words
Guardrails are not deterministic: They can fail to block something. This is part of real-world LLM security. Layer your defenses.
Security is just good engineering: If you skip security principles in your daily work, your AI implementation will inherit those vulnerabilities.
One layer is never enough: Combine prompting techniques, guardrails, structured outputs, and architectural controls. Defense in depth.
Log everything: You cannot debug what you didn’t log. Track all agent actions for explainability and auditability.
