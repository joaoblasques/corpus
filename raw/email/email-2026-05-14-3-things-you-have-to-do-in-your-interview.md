Hey Joao,

There are a few things you absolutely have to do in your coding
interview.

Otherwise, you're going to make it way harder than it needs to
be, and in a lot of cases, you're going to fail even if you
actually know the material.

So when you go through your interview, follow these 3 pointers.

1. Stream your consciousness to the interviewer
-----------------------------------------------

Most of what you're being judged on is what you say.

Your problem-solving skills, your reasoning, your tradeoffs, and
what they hear come out of your mouth.

You can make mistakes.

But you need to be very, very verbal.

If you don't say something, the interviewer doesn't know you're
thinking it.

And I can tell you, as an interviewer, it's really frustrating
when a candidate just sits there thinking and says nothing.

You have no idea where they're stuck.

You have no idea what approach they're taking.

You have no idea whether they understand the problem or are
completely lost.

More than half of the interview is just purely communication.

And even if you mess up part of the problem, if you communicate
really well, you still have a chance to move forward.

2. Code fluidly
---------------

When you get into the code editor, it needs to look like you're
actually a software engineer.

This is something I've seen a lot.

I've done tons of mock interviews with software engineers, and
when I put them into a coding environment and asked them to write
some basic Java code, it looked like they'd never written code
before.

Not because they were bad.

But because they'd gotten so used to AI tools, autocomplete, and
IDE help that they couldn't write basic syntax on their own.

They forget how to write simple loops.

They struggle to code without the editor finishing everything for
them.

It doesn't matter how good you are in the rest of the interview.
If it looks like you haven't written code before, that's a bad
signal.

So when you're doing DSA prep, practice without autocomplete.

You need to know the basic syntax well enough that you can write
it from memory.

3. Think of the interviewer like a teammate
-------------------------------------------

Don't think of the interviewer as someone who's trying to make
you fail.

Think of them like a teammate.

Communicate with them.

Ask questions.

Be friendly.

Laugh a little if it makes sense.

In later rounds especially, this person may actually be someone
who'd work with you on the team.

So yes, they're checking whether you can solve the problem.

But they're also asking themselves, "Do I want to work with this
person?"

"Would this person fit well with the team?"

That's why you need to treat the interview like you're working
with them to solve the problem.

The more you communicate, the more friendly you are, and the more
collaborative you feel, the more they're going to root for you.

You'll also get more hints because they can actually see where
you're going and how to help.

Remember: Most interviewers want you to succeed.

But they can only help you if you're communicating, being verbal,
and showing that you're someone they'd actually want to work
with.

So don't sit there silently.

Don't rely on autocomplete so much that you can't write code by
yourself.

And don't treat the interviewer like an enemy.

Keep going,

Tim
